TYPE: logical

```verus
proof fn phi_2_sorted_keys_no_duplicate_exclusion(k: CKey, v1: Vec<u8>, v2: Vec<u8>)
    requires
        v1@ != v2@,
    ensures
        !spec_sorted_keys(Vec::<CKeyKV> { inner: Ghost(seq![CKeyKV { k: k, v: v1 }, CKeyKV { k: k, v: v2 }]) }),

}
```
REASON: Two entries with the same key but different values cannot be sorted (strict `<` prevents equal keys) — but the spec implicitly forbids duplicate keys through `ckeykvlt`'s strict ordering rather than an explicit uniqueness constraint, which is fragile

NAME: ckeykvlt_ignores_value
TYPE: behavioral
CODE:
```verus
proof fn phi_3_ckeykvlt_ignores_value(k1: CKey, k2: CKey, v1: Vec<u8>, v2: Vec<u8>, v3: Vec<u8>)
    requires
        k1.ukey < k2.ukey,
    ensures
        ckeykvlt(CKeyKV { k: k1, v: v1 }, CKeyKV { k: k2, v: v2 })
            == ckeykvlt(CKeyKV { k: k1, v: v3 }, CKeyKV { k: k2, v: v2 }),
{
}
```
REASON: `ckeykvlt` compares only keys and completely ignores values — two entries with the same key but different values are considered equal by the ordering, so a sorted key-value list could silently drop value distinctions
===PHI_END===
