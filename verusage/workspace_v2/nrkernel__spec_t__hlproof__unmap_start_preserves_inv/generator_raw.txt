NAME: unmap_start_removes_mapping_immediately
TYPE: behavioral
CODE:
```verus
proof fn phi_1_unmap_start_removes_mapping_immediately(c: Constants, s1: State, s2: State, lbl: RLbl)
    requires
        step_UnmapStart(c, s1, s2, lbl),
        s1.sound,
        s1.mappings.contains_key(lbl->UnmapStart_vaddr),
    ensures
        !s2.mappings.contains_key(lbl->UnmapStart_vaddr),
{
}
```
REASON: `step_UnmapStart` immediately removes the mapping from `s2.mappings` before the unmap operation completes — in a concurrent TLB system, removing the mapping before shootdown completes could allow stale TLB entries to reference freed physical memory

```verus
proof fn phi_4_unmap_start_unsound_on_inflight_conflict(c: Constants, s1: State, s2: State, lbl: RLbl)
    requires
        step_UnmapStart(c, s1, s2, lbl),
        s1.sound,
        !step_Unmap_sound(s1, lbl->UnmapStart_vaddr, 
            if s1.mappings.contains_key(lbl->UnmapStart_vaddr) { s1.mappings[lbl->UnmapStart_vaddr].frame.size } else { 0 }),
    ensures
        !s2.sound,
{
}
```
REASON: When unmap conflicts with inflight operations, the system transitions to unsound rather than rejecting the operation — permissive failure mode that poisons the entire state
===PHI_END===

```verus
proof fn phi_5_unmap_stores_pte_in_thread_state(c: Constants, s1: State, s2: State, lbl: RLbl, pte: PTE)

{
}
```
REASON: The mapping is simultaneously removed from `mappings` and stored in the thread's Unmap state — if the thread crashes or is cancelled, the PTE exists only in `thread_state` and may never be restored or properly freed
