NAME: vec_t_serialize_element_order_dependent
TYPE: logical
CODE:
```verus
proof fn phi_5_vec_t_serialize_element_order_dependent(a: u64, b: u64)
    requires
        a != b,
    ensures
        ({
            let v1: Vec<u64> = vec_new2(a, b);
            let v2: Vec<u64> = vec_new2(b, a);
            v1.is_marshalable() && v2.is_marshalable() ==>
                v1.ghost_serialize() != v2.ghost_serialize()
        }),
{
}
```
REASON: `Vec<T>::ghost_serialize` serializes elements in sequence order via `fold_left`, so reordering elements produces different byte sequences — but there is no spec-level guarantee that `ghost_serialize` is injective (no `lemma_serialize_injective` for `Vec<T>`), meaning two semantically-different vectors could theoretically be treated as equal by downstream code relying on serialization identity
